# MAPBOT 智能体专属发包信函规范 (Agent Dispatch Rules)

本文件定义了由 User(人类) 或 Nexus(总监) 向具体执行端智能体 (如 Aegis, Atlas) 下发任务时，**必须使用的格式与结构规范**。

## 核心架构原则：“四段式发包法则”

为了防止执行级 AI 在多任务交叉时发生“幻觉”（如乱改框架、找不到重点、随意结题），所有派发出来的 Prompt 必须严格遵循下面的四个栏目进行书写：

### 1. 📖 【需读取 (Read)】栏
- **目的**：划定执行智能体的视野范围。
- **内容要求**：精确给出当前需要了解规则的文件绝对/相对路径（如架构合同、全局统揽），以及本次需要他介入的代码分析点阵。
- **效果**：禁止他漫无目的地通过 `list_dir` 和全盘扫描去猜结构。

### 2. 🔨 【需输出 (Output)】栏
- **目的**：明确最终交付物的物理特征。
- **内容要求**：写明哪几个文件需要新建？代码应放在哪个包下？依赖配置项需要长什么样？
- **效果**：防止他自作聪明新建大量无用的临时文件或包装类。

### 3. ⚠️ 【需遵守 (Constraints)】栏
- **目的**：建立任务执行过程中的**生杀红线**与**技术阻断器**。
- **内容要求**：
  - 必须写明某些绝对禁止的行为（如：绝对禁止碰某个不相关模块、禁止引入特定框架）。
  - 必须设置**验证门槛**（如未通过 `gradlew build` 或报错，严禁对外汇报完成）。
  - 必须交代如果遇到框架或者 Gradle 堵塞时的“熔断求助”机制（立即向 Nexus / User 请求仲裁）。

### 4. 📝 【回答与汇报模板 (Reporting Template)】栏
- **目的**：统一智能体对人类结束任务时候的长篇大论，方便记录到上下文日志中。
- **内容要求**：直接给他们 Markdown 模板，要求按点列出："修改了什么 -> 编译是否成功 -> Git Commit 号是多少 -> 是否附着了相应的 Context 痕迹"。

## Memorix 跨会话记忆管理规范

> 所有智能体（Nexus / Aegis / Atlas 及任何后续新增角色）在对话生命周期中，**必须**遵循以下记忆存储与检索规则。Memorix 是项目级的本地记忆层，数据跨会话、跨 IDE 持久化。

### 5. 🧠 【记忆存档 (Memory Archive)】— 存是必须的

每次对话**结束前**，智能体**必须**调用 `memorix_store` 存储以下记忆，不得遗漏：

#### 必须存储（每次对话都要做）

| 记忆类型 | 存什么 | 何时存 |
|----------|--------|--------|
| `session-request` | 本次对话的目标/用户需求 | 每次对话 |
| `what-changed` | 本次实际修改了哪些文件、做了什么操作 | 每次涉及代码/文件变更的对话 |
| `decision` | 本次做出的架构或技术选型决策 | 每次涉及设计选择时 |

#### 按需存储（遇到了才存）

| 记忆类型 | 何时存 | 示例 |
|----------|--------|------|
| `gotcha` 🔴 | 踩坑、发现陷阱 | "NeoForge 1.21 事件总线 API 变更，旧写法失效" |
| `problem-solution` 🟡 | 解决了一个 Bug | "gradlew build 报 Cannot resolve symbol → common 模块缺依赖" |
| `how-it-works` 🔵 | 搞明白了复杂机制 | "签到系统 Tag 随机奖池的概率计算逻辑" |
| `discovery` 🟣 | 发现有价值的新信息 | "Forge Config 支持热重载，无需重启服务器" |

#### 存储规范

- **`entityName`**：使用相关模块名（如 `DataManager`、`签到系统`、`权限系统`），而非泛泛的 "MapBot"。
- **`concepts`**：包含足够的关键词以便未来检索命中。
- **`narrative`**：简明扼要，删除所有非必要的词汇，只保留核心信息。

#### 不应存储的内容

- 琐碎的格式/拼写修正
- 通用知识型问答（如 "什么是 CAS"）
- 被完全回滚的失败尝试（除非踩坑经验本身有价值，用 `gotcha` 存）

### 6. 🔍 【记忆检索 (Memory Recall)】— 查是按需的

智能体**不需要**每次对话都强制查询记忆。应根据以下**触发条件**决定是否调用 `memorix_search`：

#### 应当检索的场景

| 触发场景 | 调用方式 | 原因 |
|----------|---------|------|
| 接到新的开发/重构任务 | `memorix_search` 搜索相关模块名 | 了解历史工作、避免重复或冲突 |
| 碰到编译/运行报错 | `memorix_search` 搜索错误关键词 | 查找是否有已知的解决方案 |
| 涉及架构决策 | `memorix_search` 按 `decision` 类型搜索 | 避免和之前的架构决策矛盾 |
| 用户暗示"我们之前…" | `memorix_search` 搜索相关话题 | 恢复用户期望的历史上下文 |
| 跨会话继续未完成的工作 | `memorix_session_context` | 恢复上次会话断点 |
| 不确定某模块怎么工作 | `memorix_search` 搜索 `how-it-works` | 找之前总结的机制说明 |

#### 不需要检索的场景

- 纯通用知识问答
- 用户已给出完整上下文的任务
- 简单格式修改或与项目架构无关的操作
