# MAPBOT 智能体专属发包信函规范 (Agent Dispatch Rules)

本文件定义了由 User(人类) 或 Nexus(总监) 向具体执行端智能体 (如 Aegis, Atlas) 下发任务时，**必须使用的格式与结构规范**。

## 核心架构原则：“四段式发包法则”

为了防止执行级 AI 在多任务交叉时发生“幻觉”（如乱改框架、找不到重点、随意结题），所有派发出来的 Prompt 必须严格遵循下面的四个栏目进行书写：

### 1. 📖 【需读取 (Read)】栏
- **目的**：划定执行智能体的视野范围。
- **内容要求**：精确给出当前需要了解规则的文件绝对/相对路径（如架构合同、全局统揽），以及本次需要他介入的代码分析点阵。
- **效果**：禁止他漫无目的地通过 `list_dir` 和全盘扫描去猜结构。

### 2. 🔨 【需输出 (Output)】栏
- **目的**：明确最终交付物的物理特征。
- **内容要求**：写明哪几个文件需要新建？代码应放在哪个包下？依赖配置项需要长什么样？
- **效果**：防止他自作聪明新建大量无用的临时文件或包装类。

### 3. ⚠️ 【需遵守 (Constraints)】栏
- **目的**：建立任务执行过程中的**生杀红线**与**技术阻断器**。
- **内容要求**：
  - 必须写明某些绝对禁止的行为（如：绝对禁止碰某个不相关模块、禁止引入特定框架）。
  - 必须设置**验证门槛**（如未通过 `gradlew build` 或报错，严禁对外汇报完成）。
  - 必须交代如果遇到框架或者 Gradle 堵塞时的“熔断求助”机制（立即向 Nexus / User 请求仲裁）。

### 4. 📝 【回答与汇报模板 (Reporting Template)】栏
- **目的**：统一智能体对人类结束任务时候的长篇大论，方便记录到上下文日志中。
- **内容要求**：直接给他们 Markdown 模板，要求按点列出：“修改了什么 -> 编译是否成功 -> Git Commit 号是多少 -> 是否附着了相应的 Context 痕迹”。
