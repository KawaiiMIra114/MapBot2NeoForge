> 约束：权限仅 3 级：`user`、`admin`、`owner`。

## 1. 设计目标
- 易读：主命令少、别名保留、帮助菜单按场景自动收敛。
- 易上手：玩家默认只看到自己能用的命令。
- 高安全：高危命令仅 `owner`，并要求二次确认与审计。
- 可维护：权限判断只走一条规则，不再混用多套语义。

## 2. 权限模型

### 2.1 角色定义
| 角色 | 说明 | 典型人群 |
|---|---|---|
| `user` | 普通玩家，只能执行自助/查询/娱乐命令 | 玩家 |
| `admin` | 服务器管理员，负责日常运维与玩家管理 | 服管 |
| `owner` | 机器人开发者/系统负责人，负责权限治理与高危操作 | Bot 开发者 |

### 2.2 权限边界（关键）
- `user` 不可执行任何会影响他人状态或服务器状态的命令。
- `admin` 可以做玩家管理，但不能做权限体系治理与全局高危操作。
- `owner` 才能执行：权限变更、管理员变更、关服链路等高危命令。

### 2.3 禁止事项
- `admin` 禁止：提升他人到 `owner`、修改核心鉴权策略、绕过审计执行高危命令。
- `user` 禁止：执行任何管理命令。

## 3. 命令体系总览

### 3.1 规范原则
- 每个能力只保留 1 个主命令名。
- 历史命令保留为兼容别名，但在帮助中只突出主命令。
- 同类命令参数风格统一（先对象后选项，或先选项后对象，不能混乱）。

### 3.2 建议分类
- 功能类：绑定、状态、在线、时长、帮助。
- 娱乐类：签到、领奖、兑换码。
- 管理类：查背包、查位置、禁言、解禁、强制解绑、重载。
- 核心治理类（Owner）：权限管理、管理员管理、关服控制。

## 4. 新命令与权限矩阵

> 说明：以下为“主命令 + 兼容别名”设计稿。  
> 风险等级：`低/中/高`（是否影响他人、是否影响全服）。

| 主命令 | 兼容别名 | 分类 | 最低权限 | 可用场景 | 用途 | 风险 | 状态 |
|---|---|---|---|---|---|---|---|
| `#help` | `#菜单` | 功能 | user | 玩家群/管理群/私聊 | 显示当前场景可用命令 | 低 | 保留 |
| `#id` | `#bind` `#绑定` | 功能 | user | 玩家群/私聊 | 绑定 QQ 与游戏账号 | 低 | 保留 |
| `#unbind` | `#解绑` | 功能 | user | 玩家群/私聊 | 解除自身绑定 | 低 | 保留 |
| `#list` | `#在线` | 功能 | user | 玩家群/私聊 | 按服务器分组显示在线玩家 | 低 | 保留 |
| `#status` | `#状态` `#tps` | 功能 | user | 玩家群/私聊 | 查看服务器状态 | 低 | 保留 |
| `#time` | `#playtime` `#在线时长` `#时间` | 功能 | user | 玩家群/私聊 | 查询在线时长（自己/管理员可查他人） | 低 | 合并（统一主命令） |
| `#sign` | `#签到` | 娱乐 | user | 玩家群/私聊 | 每日签到抽奖 | 低 | 保留 |
| `#accept` | `#领取` | 娱乐 | user | 玩家群/私聊 | 领取签到奖励 | 低 | 保留 |
| `#cdk` | `#兑换码` | 娱乐 | user | 私聊优先（群内仅提示） | 获取兑换码 | 中 | 保留（强化隐私） |
| `#inv` | - | 管理 | admin | 管理群 | 查询玩家背包/末影箱 | 中 | 保留 |
| `#location` | `#位置` | 管理 | admin | 管理群 | 查询玩家坐标 | 中 | 保留 |
| `#mute` | `#禁言` | 管理 | admin | 管理群 | 禁言玩家 | 中 | 保留 |
| `#unmute` | `#解禁` | 管理 | admin | 管理群 | 解除禁言 | 中 | 保留 |
| `#adminunbind` | `#强制解绑` | 管理 | admin | 管理群 | 强制解绑指定 QQ | 中 | 保留 |
| `#agreeunbind` | - | 管理 | admin | 管理群 | 冲突仲裁解绑 | 中 | 保留（可并入上条） |
| `#reload` | - | 管理 | admin | 管理群 | 重载 Alpha + 子服配置 | 中 | 保留 |
| `#setperm` | - | 治理 | owner | 管理群/owner私聊 | 设置角色（user/admin/owner） | 高 | 保留（Owner 独占） |
| `#addadmin` | - | 治理 | owner | 管理群/owner私聊 | 添加 admin | 高 | 建议收敛到 `#setperm` |
| `#removeadmin` | - | 治理 | owner | 管理群/owner私聊 | 移除 admin | 高 | 建议收敛到 `#setperm` |
| `#stopserver` | `#关服` | 治理 | owner | 管理群/owner私聊 | 发起关服倒计时 | 高 | 保留（二次确认） |
| `#cancelstop` | `#取消关服` | 治理 | owner | 管理群/owner私聊 | 取消关服计划 | 高 | 保留（二次确认） |

### 4.1 默认开放与默认隐藏
- 默认开放给 `user`：`help id unbind list status time sign accept cdk`。
- 默认隐藏高危集（仅 `owner` 可见）：`setperm addadmin removeadmin stopserver cancelstop`。

## 5. 场景与可见性规则
- 玩家群：仅显示 `user` 命令。
- 管理群：显示 `admin + owner` 命令（按权限过滤）。
- 私聊：默认只显示该用户可用命令；敏感返回优先私聊。
- `#help all`：显示“本角色可见的全部命令”，不显示越权命令。

## 6. 安全基线
- 白名单群限制：命令仅在配置允许的群内执行。
- 高危二次确认：`owner` 执行高危命令前需二次确认。
- 审计日志：记录时间、操作者、来源、命令、结果、追踪ID。
- 速率限制：单用户短时连续调用触发限流。
- 错误回显最小化：群内只返回摘要，不暴露内部细节。
- 敏感数据私聊化：兑换码/密钥类信息仅私聊下发。

## 7. 上手流程（低门槛）

### 7.1 新玩家
1. 发送 `#help` 看可用命令。  
2. 发送 `#id 玩家名` 完成绑定。  
3. 日常用 `#list`、`#status`、`#time`。  
4. 签到链路：`#sign -> #accept`，离线用 `#cdk`。

### 7.2 新服管（admin）
1. 在管理群先执行 `#myperm` 确认角色。  
2. 玩家管理用 `#mute/#unmute/#adminunbind`。  
3. 运维操作用 `#reload`。  
4. 涉及权限与关服，交由 `owner`。

### 7.3 Owner
1. 仅在必要时执行高危命令。  
2. 权限治理优先使用 `#setperm`（避免多命令分叉）。  
3. 关服链路必须做二次确认并留审计记录。

## 8. 迁移与兼容策略
- 第一步：保留旧别名，帮助菜单只推荐主命令。
- 第二步：将权限判定统一成 3 级角色模型。
- 第三步：把 `addadmin/removeadmin` 逐步收敛到 `setperm`。
- 第四步：稳定后发布“精简命令清单”，保留兼容别名至少 1 个版本周期。

## 9. 待你确认的决策点
1. 是否将 `#reload` 保持为 `admin` 可执行（当前建议：是）。
2. `#agreeunbind` 是否并入 `#adminunbind`（减少心智负担）。
3. 是否正式废弃 `#playtime`，只保留为兼容别名。
4. `owner` 私聊是否允许执行全部高危命令（当前建议：允许，但必须审计）。
