## 1. 目标与范围
本清单将系统重构拆解为按开发生命周期推进的连续步骤，不使用 `P0-P4` 分组，而采用“一个可交付系统从定义到发布”的顺序。

覆盖范围：
- Alpha 控制面
- Reforged 执行面
- Bridge 协议面
- 数据一致性与可观测
- 运维、迁移、发布与开源治理

## 2. 为什么按这个顺序
采用该顺序的原因：
1. 先冻结规则再改代码，避免实现与合同反复打架。
2. 先做最难回滚的基础层（协议/一致性/安全），再做上层功能，减少返工。
3. 每阶段都可独立验收，保证中途可停、可回滚、可审计。
4. 先建立测试与门禁，再放大改动范围，降低“改完才发现不可用”的风险。

## 2.1 单人维护模式说明
本项目按“单人长期维护”设计执行，治理规则如下：
1. 无需固定月度/季度维护节奏，采用事件驱动维护。
2. 以下事件发生时必须维护：重构进行中、功能制作过程、重大安全事件。
3. 所有“评审/签署”语义统一为“自审 + 自记录”，避免依赖多人流程。
4. 任何阶段都允许暂停与续做，但恢复时必须先补齐文档与基线记录。

## 3. 开发顺序任务清单

## 阶段A：重构立项与基线冻结
### A1 立项冻结
- 输入：`SYSTEM_CRITIQUE_AND_REFACTOR_PAPER_V1.md`、全部新合同文档。
- 任务：
1. 冻结当前重构范围、非目标、完成定义（DoD）。
2. 指定单一决策责任人（你本人）与“不可用时冻结高风险变更”机制（按 ADR 模板）。
3. 建立“变更必须先改合同”的评审门禁。
- 产出：重构 Charter、DoD、评审门禁说明。
- 验收标准：
1. 每项重构任务都可映射到合同条目。
2. 无“口头约定”未落文档条目。

### A2 基线采样
- 输入：当前运行环境与日志。
- 任务：
1. 记录基线指标（延迟、超时率、重试率、注册失败率）。
2. 固定基线配置快照（Alpha + Reforged + Redis + OneBot）。
3. 固定基线行为快照（`#bind/#unbind/#status/#list/#reload`）。
- 产出：Baseline 数据包（配置、指标、行为报告）。
- 验收标准：可用于重构后逐项对比。

## 阶段B：契约层落地与代码映射
### B1 Bridge 消息与错误契约映射
- 输入：`BRIDGE_MESSAGE_CONTRACT.md`、`BRIDGE_ERROR_CODE_CONTRACT.md`。
- 任务：
1. 建立消息字段映射表（implemented/planned/deprecated -> 代码位置）。
2. 实现结构化错误码双栈映射（保留过渡窗口）。
3. 实现消息体大小门禁（64KiB 单帧 + 超限错误码）。
- 产出：协议映射清单、错误码映射清单、实现差距清单。
- 验收标准：
1. 合同字段与代码映射无缺口。
2. 所有错误回包可解析为标准错误码。

### B2 权限与配置契约映射
- 输入：`COMMAND_AUTHORIZATION_CONTRACT.md`、`CONFIG_SCHEMA_CONTRACT.md`。
- 任务：
1. 统一授权口径（仅 `user/admin/owner`）。
2. 配置 schema 校验器落地（类型、范围、热重载、回滚）。
3. 越权与配置变更审计字段统一。
- 产出：授权与配置校验模块改造方案。
- 验收标准：
1. 未定义角色无法执行任何命令。
2. 非法配置热重载 100% 回滚。

### B3 一致性与SLO契约映射
- 输入：`DATA_CONSISTENCY_CONTRACT.md`、`OBSERVABILITY_SLO_CONTRACT.md`。
- 任务：
1. 明确实体事实源与版本字段。
2. 确认 CAS 冲突返回与审计上报。
3. 确认 SLI/SLO 指标采集点与告警级别实现位点。
- 产出：一致性状态机与指标字典。
- 验收标准：每个合同指标都能在代码里找到采集入口。

## 阶段C：架构硬化设计评审
### C1 线程模型与故障模型评审
- 输入：`THREADING_MODEL.md`、`FAILURE_MODEL.md`。
- 任务：
1. 明确主线程/IO线程/Worker 的访问白名单与黑名单。
2. 定义断连后 pending 请求生命周期。
3. 定义半成功补偿策略。
- 产出：线程边界整改清单、故障状态机图。
- 验收标准：
1. 所有高风险路径有整改责任（你本人）与计划完成时点。
2. 故障状态迁移可被日志重建。

### C2 安全边界与版本兼容评审
- 输入：`SECURITY_BOUNDARY.md`、`VERSIONING_AND_COMPATIBILITY.md`。
- 任务：
1. 定义密钥轮换 SOP 与回滚点。
2. 定义协议版本协商与弃用窗口。
3. 定义匿名与鉴权路由边界。
- 产出：安全与版本治理实施蓝图。
- 验收标准：敏感接口全部有鉴权规则与日志审计规则。

## 阶段D：核心代码重构（底层能力）
### D1 Bridge 通道重构
- 任务：
1. 引入 `protocol_version` 与能力协商。
2. 引入变更型请求幂等键去重缓存（含 TTL 与冲突语义）。
3. 引入断连快速失败与 pending 统一回收。
- 产出：新 Bridge 基础通信能力。
- 验收标准：
1. 无幂等键冲突误判。
2. 断连后请求不会长时间堆积。

### D2 线程与执行模型重构
- 任务：
1. 网络线程只读快照，不直接访问游戏对象。
2. 所有游戏副作用操作统一回主线程。
3. 去除高风险阻塞调用（主线程 `get/join/sleep`）。
- 产出：线程安全改造提交集。
- 验收标准：线程越界扫描与压测通过。

### D3 数据一致性与恢复重构
- 任务：
1. 写入改造为版本化 CAS。
2. 实现回放与补偿流程。
3. 统一原子持久化策略（tmp + 原子替换 + 备份）。
- 产出：一致性引擎与恢复流程实现。
- 验收标准：
1. 并发冲突不再静默覆盖。
2. 重启后可恢复到一致状态。

## 阶段E：业务层重构（命令与流程）
### E1 命令语义统一
- 任务：
1. 统一命令权限判定链路。
2. 清理命令重复语义并保留兼容别名。
3. 统一帮助菜单分场景输出（玩家群/管理群/私聊）。
- 产出：命令系统一致化版本。
- 验收标准：同一命令在不同入口不再出现冲突判定。

### E2 关键业务链路重构
- 任务：
1. 绑定链路改为“解析身份 -> 写权威 -> fan-out 子服 -> 回执聚合”。
2. 解绑链路补齐全服白名单移除。
3. `switch_server` 改造为强回执语义，不允许假成功。
- 产出：关键链路端到端一致行为。
- 验收标准：关键链路每一步都有可观测回执与失败分类。

### E3 管理面API语义统一
- 任务：
1. `/api/servers/{id}/command` 改为等待执行回执或明确 pending。
2. 统一 API 与 QQ 命令回执语义。
3. 统一错误码输出格式。
- 产出：控制面接口一致行为模型。
- 验收标准：API 与机器人入口对同一动作返回一致语义。

## 阶段F：可观测与运维能力落地
### F1 指标与告警落地
- 任务：
1. 按 SLO 合同接入核心指标。
2. 建立 Warning/Critical 双级告警规则。
3. 建立最小证据集自动采集模板。
- 产出：可观测仪表盘与告警策略。
- 验收标准：故障发生时 10 分钟内可定位到模块级根因范围。

### F2 运维手册联调
- 任务：
1. 用 `DEPLOYMENT_RUNBOOK` 执行全新部署演练。
2. 用 `OPERATIONS_RUNBOOK` 执行 `#reload` 正向/负向验证。
3. 用 `INCIDENT_RESPONSE_PLAYBOOK` 执行一次故障演练。
- 产出：运维实操验收记录。
- 验收标准：新人按手册可独立完成操作且结果一致。

## 阶段G：测试体系与质量门禁
### G1 契约测试与集成测试
- 任务：
1. 将 Bridge 合同测试用例接入自动化。
2. 建立 Alpha-Reforged 端到端测试。
3. 增加断连、乱序、超时、重放的故障注入测试。
- 产出：自动化测试套件。
- 验收标准：关键链路至少覆盖成功/失败/超时/重试四类场景。

### G2 发布门禁自动化
- 任务：
1. 将 `RELEASE_CHECKLIST` 转换为 CI 门禁任务。
2. 增加文档一致性与术语漂移检测。
3. 增加安全扫描（敏感信息、明文 token、错误码漂移）。
- 产出：可阻断的质量流水线。
- 验收标准：未通过门禁无法发布。

## 阶段H：迁移、灰度与回滚
### H1 升级迁移演练
- 任务：
1. 按 `UPGRADE_MIGRATION_GUIDE` 执行预演。
2. 演练“灰度成功但全量失败”的回滚路径。
3. 验证数据一致性与业务可用性在回滚后恢复。
- 产出：迁移演练报告。
- 验收标准：回滚时间与数据恢复满足手册阈值。

### H2 生产灰度发布
- 任务：
1. 分阶段发布（暗发布 -> 小流量 -> 中流量 -> 全量）。
2. 每阶段对照基线做差异评估。
3. 不达标则回滚并形成 RCA。
- 产出：灰度发布记录与阶段评审记录。
- 验收标准：全量后稳定窗口内指标达到合同目标。

## 阶段I：稳定化与开源发布准备
### I1 稳定化冲刺
- 任务：
1. 清理灰度遗留问题与技术债。
2. 完成合同-实现-手册三方一致性复审。
3. 更新指标阈值与运行基线。
- 产出：稳定版候选分支（RC）。
- 验收标准：连续稳定窗口无 Sev-1/Sev-0 事故。

### I2 开源治理落地
- 任务：
1. 补齐 LICENSE、CONTRIBUTING、SECURITY、CODE_OF_CONDUCT。
2. 补齐示例配置（去敏）。
3. 固化发布产物结构与对外文档入口。
- 产出：可发布的开源包。
- 验收标准：外部贡献者可按文档完成构建、测试、运行。

## 阶段J：收尾与长期治理
### J1 复盘与知识沉淀
- 任务：
1. 输出重构复盘（做对/做错/待优化）。
2. 把关键决策沉淀为 ADR。
3. 更新 `INDEX` 与 `GLOSSARY`，防止知识漂移。
- 产出：长期治理包。
- 验收标准：后续版本可按同一流程复用。

### J2 常态化机制
- 任务：
1. 建立事件触发的合同一致性巡检（重构中/开发变更后/重大安全事件后）。
2. 建立事件触发的故障演练（发布前、关键链路改造后、事故复盘后）。
3. 建立事件触发的版本兼容与弃用审计（协议变更前后、迁移演练前后）。
- 产出：持续治理日历。
- 验收标准：系统进入“可持续演进”状态，而非一次性修补。

## 4. 全流程里程碑验收门槛
每个阶段结束前都必须同时满足：
1. 文档已更新并完成自审记录。
2. 代码实现已通过对应自动化测试。
3. 运维步骤可复现且有日志证据。
4. 回滚路径已演练且可执行。

## 5. 使用说明
1. 该清单是“开发顺序主轴”，不是优先级列表。
2. 若出现紧急线上风险，可临时插入修复任务，但必须回写到对应阶段与合同。
3. 阶段切换必须有明确的自审结论与变更记录（日期、原因、结果）。
