# INCIDENT_RESPONSE_PLAYBOOK

## 1. 目标与范围
- 目标：在事故发生时，用统一流程将 MTTR（平均恢复时间）降到最低，并保证过程可审计、可复盘。
- 范围：MapBot 服务可用性、功能正确性、配置与权限异常、外部依赖异常。
- 非范围：需求变更、非紧急性能优化、长期重构议题（转 ADR）。

## 2. 角色与职责
| 角色 | 职责 | 备注 |
|---|---|---|
| Incident Commander (IC) | 定级、指挥、升级、宣布恢复 | 每次事故仅 1 人 |
| Operator | 执行止血与回滚动作 | 可多人并行 |
| Investigator | 排障定位、根因分析 | 与 Operator 分离更佳 |
| Comms | 对内外同步进展 | 固定 15 分钟节奏 |
| Recorder | 记录时间线与证据 | 复盘输入源 |

## 3. 事故分级标准（强制）
| 级别 | 触发条件 | 响应时限 | 升级路径 | 退出条件 |
|---|---|---|---|---|
| Sev-0 | 全量不可用 >= 5 分钟；关键数据损坏；确认安全入侵或高危泄露 | 5 分钟内 IC 拉起 War Room 并广播 | IC -> Tech Owner -> 项目负责人 -> 全员应急 | 服务恢复并稳定 >= 60 分钟；数据完整性校验通过；高危入口完成封堵 |
| Sev-1 | 核心功能不可用，影响 >50% 用户；核心命令失败率 >20% 且持续 10 分钟 | 10 分钟内完成定级与首轮止血 | IC -> Tech Owner；15 分钟未止血升级项目负责人 | 关键链路恢复；错误率回落至基线 +20% 内并稳定 >= 45 分钟 |
| Sev-2 | 非核心功能异常，影响 10%-50% 用户；有可用降级路径 | 30 分钟内响应并给出缓解方案 | On-Call -> IC；30 分钟未缓解升级 Sev-1 流程 | 影响面降至 <10%；降级或修复后稳定 >= 30 分钟 |
| Sev-3 | 轻微异常或可绕过问题；影响 <10% 用户且无扩散迹象 | 4 小时内响应 | On-Call 处理；24 小时内重复 3 次升级 Sev-2 | 问题可复现并有修复单；临时规避方案生效且 24 小时无重复告警 |

退出规则补充：
- 任何级别若在观察窗口内复发，立即恢复到上一高等级并重新计时。
- 未满足退出条件不得宣布“已恢复”。

## 4. 响应流程（强制）
### 4.1 发现与宣告
1. 触发来源：监控告警、用户反馈、日志异常、值班巡检。
2. IC 在 5 分钟内完成定级并创建事故频道（或线程）。
3. 记录首条消息：事故编号、时间、级别、影响面、当前负责人。

### 4.2 止血（Containment）
按“先恢复服务，再追根因”执行，至少完成一项：
- 回滚到最近稳定版本。
- 关闭故障开关（feature flag / command gate）。
- 降级非核心模块（限流、熔断、只读模式）。
- 隔离异常依赖（断开第三方、切备用配置）。

止血动作矩阵：
| 场景 | 首选动作 | 次选动作 | 验证信号 |
|---|---|---|---|
| 发布后故障 | 回滚版本 | 关闭新特性 | 错误率下降、告警恢复 |
| 配置失效 | 恢复基线配置 | 切只读/降级 | 配置校验通过 |
| 权限异常 | 启用最小权限策略 | 临时封禁高危命令 | 审计日志无拒绝风暴 |
| 外部依赖异常 | 切备用端点 | 本地缓存兜底 | 超时率下降 |

### 4.3 排障（Diagnosis）
1. 固化现场：导出日志、配置快照、版本信息、最近变更记录。
2. 形成假设：按“代码变更/配置变更/依赖变更/数据变更”四类排查。
3. 二分验证：一次仅改一个变量，避免混合操作。
4. 记录证据：每次实验记录“操作-结果-结论”。
5. 归因确认：满足“可复现 + 可解释 + 可验证修复”。

### 4.4 恢复与关闭
1. 服务指标恢复到基线窗口（建议连续 30 分钟）。
2. IC 宣布“已恢复”，同步后续观察期和责任人。
3. 进入复盘流程并锁定事故记录，不得补写关键事实。

## 5. 沟通模板（强制）
### 5.1 首次通报模板
```text
[INCIDENT][Sev-X][进行中]
编号: INC-YYYYMMDD-XX
发现时间: <UTC+8 hh:mm>
影响范围: <功能/用户比例>
当前动作: <止血措施>
下次同步: <hh:mm>
负责人: <IC/Operator>
```

### 5.2 进展同步模板
```text
[INCIDENT][Sev-X][更新 N]
当前状态: <扩散/稳定/恢复中>
已完成: <动作与结果>
待执行: <下一步>

```

### 5.3 结案模板
```text

最终动作: <修复/回滚/配置恢复>
复盘时间: <YYYY-MM-DD hh:mm>
```

## 7. 验收标准
- 事故记录完整（定级、动作、时间线、责任人）。
- 至少一条可执行防再发行动项已创建并分配 Owner。
- 复盘在恢复后 48 小时内完成并归档。

## 8. 回滚与应急
- 若修复方案在观察期内导致二次故障，立即执行最近一次稳定版本回滚。
- 回滚后仍异常，切换到最小可用模式并升级至 Sev-0 处理。

## 9. 记录与审计
- 事故编号规则：`INC-YYYYMMDD-序号`。
- 必存证据：日志片段、配置 diff、发布版本号、回滚记录、沟通记录链接。
- 存档位置：`Project_Docs/Reports/`（复盘报告）与工单系统（执行跟踪）。

## 10. 证据采集模板（最小字段，强制）
使用说明：
- 事故发生后 15 分钟内必须先填“必须字段”。
- 可选字段按现场可得信息补齐，不得阻塞止血动作。

### 10.1 必须字段（20 项）
| 字段 | 级别 | 说明 | 示例 |
|---|---|---|---|
| incident_id | 必须 | 事故唯一编号 | `INC-20260214-01` |
| severity | 必须 | 事故级别 | `Sev-1` |
| status | 必须 | 当前状态 | `进行中/已恢复` |
| detection_time | 必须 | 首次发现时间 | `2026-02-14 10:03 UTC+8` |
| declare_time | 必须 | IC 宣告时间 | `2026-02-14 10:06 UTC+8` |
| reporter | 必须 | 发现人 | `oncall_a` |
| incident_commander | 必须 | 事故指挥 | `ic_b` |
| operator_owner | 必须 | 操作执行负责人 | `op_c` |
| impacted_scope | 必须 | 影响功能/模块 | `消息桥接` |
| impacted_users_ratio | 必须 | 影响用户占比 | `55%` |
| trigger_signal | 必须 | 触发信号 | `错误率告警` |
| current_version | 必须 | 事故时版本 | `v5.2.1` |
| recent_change_ref | 必须 | 最近变更引用 | `commit abc123` |
| containment_action | 必须 | 已执行止血动作 | `关闭 feature flag` |
| containment_start_time | 必须 | 止血开始时间 | `10:08` |
| containment_result | 必须 | 止血结果 | `错误率下降` |
| rollback_decision | 必须 | 是否回滚与决策人 | `No, IC` |
| recovery_time | 必须 | 宣告恢复时间 | `10:52` |
| primary_root_cause | 必须 | 一句话根因 | `配置键缺失` |
| evidence_bundle_link | 必须 | 证据包链接 | `工单/对象存储链接` |
