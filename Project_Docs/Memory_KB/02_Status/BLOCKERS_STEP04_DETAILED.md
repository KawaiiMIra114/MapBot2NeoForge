# Step-04 Blockers Detailed (B2 -> B3)

更新时间：2026-02-15
结论：当前 4 个阻断均为 `P0`（放行阻断）；`P0` 未清零即 `NO-GO B3`。

## 优先级分层

| 阻断项 | 优先级 | 是否可延后 | 说明 |
|---|---|---|---|
| B-01 角色模型未收敛 | P0 | 否 | 角色语义不一致会直接导致授权判断偏差 |
| B-02 AUTH-403 未统一 | P0 | 否 | 拒绝路径语义不统一会造成调用端误判成功 |
| B-03 配置 fail-closed 缺失 | P0 | 否 | 非法配置可进入运行态，违反安全边界 |
| B-04 reload 事务闭环缺失 | P0 | 否 | 热重载半提交会破坏一致性与可恢复性 |

可延后项（P1）仅限增强类工作，不属于当前阻断闭环范围，例如：审计字段扩展、错误文案优化、观测面板可视化。

## B-01 角色模型未收敛（VIEWER/OPERATOR/ADMIN -> user/admin/owner）
- 现状：系统仍使用 `VIEWER/OPERATOR/ADMIN`；`owner` 语义缺失。
- 目标：统一到合同角色 `user/admin/owner`，并对未知角色显式拒绝。
- 差距：角色枚举、映射与鉴权判定链路未按合同收敛。
- 最小修复：
  1. 引入 `ContractRole{user,admin,owner}` 作为唯一鉴权输入。
  2. 建立 legacy->contract 映射：`VIEWER->user`、`OPERATOR->admin`、`ADMIN->owner`（若合同另有定义，以合同为准）。
  3. 未知/空角色统一走拒绝分支（`HTTP 403 + AUTH-403`）。
- 验收口径：
  1. 全链路不再出现 legacy 角色作为决策输入。
  2. `owner` 专属能力可测且与 `admin` 有边界。
  3. 未知角色请求全部返回 `403/AUTH-403`，且零副作用。
- 风险：角色映射错误会造成“权限扩大”或“误拒绝”。

## B-02 AUTH-403 未统一（/api/users*、/api/mapbot*）
- 现状：存在 `HTTP 200 + 错误文本` 的拒绝路径。
- 目标：越权拒绝统一为 `HTTP 403` 且 `errorCode=AUTH-403`。
- 差距：接口实现与错误映射未统一，仍有旁路返回成功状态码。
- 最小修复：
  1. 抽取统一鉴权拒绝响应构造器（status/body/header 一致）。
  2. 扫描并替换 `/api/users*`、`/api/mapbot*` 所有越权分支。
  3. 增加回归用例：逐接口校验“越权=403/AUTH-403”。
- 验收口径：
  1. 越权请求 100% 返回 `403/AUTH-403`。
  2. 不再出现 `200 + 错误文本` 的拒绝响应。
  3. 拒绝路径不产生写入、副作用或状态漂移。
- 风险：遗漏单条路径会导致调用方按成功分支继续执行。

## B-03 配置 fail-closed 缺失（alpha.properties）
- 现状：未知键/非法值可 fallback，进程可继续运行。
- 目标：unknown-key fail-closed；类型/范围校验失败即拒绝加载。
- 差距：配置解析缺 schema 强校验与硬失败策略。
- 最小修复：
  1. 定义 `alpha.properties` 白名单 schema（键名、类型、范围、必填）。
  2. 加载阶段执行严格校验；任一失败立即中止本次加载。
  3. 输出可审计错误（键名、期望类型/范围、实际值）。
- 验收口径：
  1. unknown-key 必然失败，不允许 fallback。
  2. 非法值（类型/范围）必然失败，不进入运行态。
  3. 合法配置加载成功且与失败路径可区分审计。
- 风险：若仍允许 fallback，非法配置会静默潜入生产路径。

## B-04 reload 事务闭环缺失（#reload）
- 现状：reload 非事务式；缺“解析->校验->原子替换->审计->失败回滚”闭环证据。
- 目标：reload 全流程事务化，失败可回滚到上一个已知好版本。
- 差距：缺 staging、atomic swap、rollback 与演练证据。
- 最小修复：
  1. 新配置先进入 staging（只读校验，不影响现网）。
  2. 校验通过后执行原子替换；失败则保持旧配置不变。
  3. 全程落审计日志（版本、操作者、结果、回滚点）。
  4. 提供一次成功演练 + 一次失败回滚演练证据。
- 验收口径：
  1. 成功路径：新配置生效且配置版本单调递增。
  2. 失败路径：系统状态与旧配置完全一致（可复核）。
  3. 审计记录完整可追溯（请求、校验、替换/回滚结果）。
- 风险：半提交会导致运行时配置漂移，且故障恢复不可预测。
