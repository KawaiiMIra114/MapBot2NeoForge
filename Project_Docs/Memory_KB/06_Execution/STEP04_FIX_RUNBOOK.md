# STEP04 Fix Runbook

## 目标与约束
- 目标：将 Step-04（B2）从 FAIL 推进到可复核状态，并给出可审计结论。
- 强约束：默认不构建；构建与打包由主控侧统一验证。
- 范围：仅修复与 Step-04 阻断项直接相关的问题，避免无关重构。

## 执行顺序（必须按序）
1. 读取上下文
- 必读：
  - `Project_Docs/Memory_KB/README.md`
  - `Project_Docs/Memory_KB/02_Status/CURRENT_STATE.md`
  - `Project_Docs/Memory_KB/02_Status/BLOCKERS_STEP04.md`
  - `Project_Docs/Memory_KB/03_Plan/NEXT_ACTIONS.md`
- 提取：当前判定、阻断项清单、验收标准、已有证据缺口。
- 产出：本轮修复任务清单（逐条映射阻断项）。

2. 定位代码
- 基于阻断项逐条定位：入口层、鉴权层、配置解析层、事务与审计链路。
- 对每个问题记录：
  - 触发路径（输入 -> 处理 -> 输出）
  - 当前行为与期望行为差异
  - 最小修改点（文件/函数/分支）
- 要求：先确认根因再改代码，不做猜测式修补。

3. 实施修复
- 按阻断项一条条落地，保证每次改动可解释、可回滚。
- 修复时保持：
  - 鉴权语义一致（含角色收敛与越权统一返回）
  - 配置 fail-closed
  - reload 全链路闭环（parse/validate/staging/atomic/audit/rollback）
- 禁止：顺手改风格、大面积重命名、跨模块无关变更。

4. 本地验证（不构建）
- 仅执行轻量、非构建型验证：
  - 静态检查（可用时）
  - 定向单测/集成用例（可直跑时）
  - 最小复现脚本或请求回放
- 明确记录：
  - 已验证项
  - 未验证项及原因
- 再次强调：默认不执行构建/打包；由主控侧完成最终构建验证。

5. 回填文档与证据
- 更新 Step-04 对应状态与行动文档（按 Memory_KB 约定位置）。
- 写清：
  - 根因
  - 修复动作
  - 验证结果
  - 剩余风险与待主控验证项
- 证据要求：命令、关键输出摘要、结论可追溯。

6. 门禁判定
- 按 Step-04 门禁项逐条打勾，不允许“整体感觉通过”。
- 结论必须二选一：
  - `GO`：全部阻断关闭，且证据完整
  - `NO-GO`：任一阻断未关闭或证据不足
- 若 `NO-GO`：给出最小下一步闭环动作（负责人/输入/完成标准）。

7. 输出结论
- 固定输出结构：
  - Verdict
  - Blocking Issues（已关/未关）
  - Fix Actions
  - Verification Scope（已验/未验）
  - Build Validation（主控侧执行）
  - Next Actions

## 执行纪律
- 不覆盖他人改动，不回退无关变更。
- 仅在 Step-04 问题域内修改，保持最小影响面。
- 所有结论必须能被文档与证据反向追踪。
