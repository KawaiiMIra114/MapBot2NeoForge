## 1. 事件触发模型

### 1.1 核心触发事件

| 事件ID | 事件名称 | 触发条件 | 必执行动作 | 时限 |
|---|---|---|---|---|
| EVT-01 | 重构执行中 | 任何 RE_STEP 进入 RUNNING | 合同一致性巡检 + 编译验证 | 每步完成前 |
| EVT-02 | 功能变更后 | 新功能合并到 main 分支 | 回归测试 + 冒烟测试 + 门禁校验 | 合并后 24h |
| EVT-03 | 重大安全事件 | Sev-0/Sev-1 事故发生 | 止血 → 诊断 → RCA → 防再发行动 | 72h 内闭环 |
| EVT-04 | 发布前 | 版本 tag 创建前 | 全量门禁 + 灰度评估 + Go/No-Go | 发布前 48h |
| EVT-05 | 依赖升级 | MC/NeoForge/JDA 版本变更 | 兼容审计 + 迁移演练 | 升级前完成 |
| EVT-06 | 定期巡检 | 每月 1 日 | INDEX/GLOSSARY 可达性 + 合同漂移扫描 | 月初 7 天内 |

### 1.2 未触发执行的阻断后果

| 事件 | 阻断后果 |
|---|---|
| EVT-01 未执行 | 当前 Step Verdict=FAIL，禁止推进 |
| EVT-02 未执行 | 禁止创建 Release Tag |
| EVT-03 未执行 | 挂起所有发布，直到 RCA 完成 |
| EVT-04 未执行 | 禁止发布 |
| EVT-05 未执行 | 禁止升级依赖版本 |
| EVT-06 未执行 | 标记 P1 文档债务 |

### 1.3 最小证据要求

每次触发必须留存：
1. 触发时间戳 (UTC ISO-8601)
2. 触发事件 ID
3. 执行动作日志 (`.log` + `.exit`)
4. 结论与后续行动 (verdict)

## 2. 治理策略执行模式

```
事件发生 → 触发检测 → 执行动作 → 留存证据 → 结论判定
    ↓                                         ↓
 未触发 → 阻断后果生效                结论不通过 → 升级处理
```

## 3. 单人维护适配
- 所有动作可由单人执行（自审+自记录）
- 无需外部审批，但需留存可审计证据
- 定期巡检 (EVT-06) 可合并到版本发布周期
